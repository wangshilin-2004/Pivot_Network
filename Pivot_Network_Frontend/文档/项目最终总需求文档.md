# 项目最终总需求文档

更新时间：`2026-05-10`

## 1. 文档目的

本文面向当前主目录下的两个核心子目录：

- `Pivot_Network_Backend-main/`
- `haihe_-main/`

在统一阅读两部分代码、说明文档和需求材料后，对整体项目的业务定位、建设目标、功能范围、技术边界、当前现状和后续推进方向进行综合梳理，形成一份可用于项目汇报、交接说明和后续实施的总需求文档。

本文强调“结合现有代码现实进行反推”，不将尚未证实的架构关系写成既定事实。

## 2. 总体结论

从当前目录结构和代码实现看，`Pivot_Network_Backend-main` 与 `haihe_-main` 并不是一个已经完成统一集成的单仓系统，当前也没有发现明确的直接代码依赖、统一构建关系或统一部署编排关系。

但从业务语义和模块边界看，两部分具备较强的互补性，可被归纳为同一大项目的两个子方向：

- `Pivot_Network_Backend-main`：偏“算力交易、运行时会话编排、安全访问”的后端与客户端原型
- `haihe_-main`：偏“平台运营管理、设备管理、规则告警、权限后台”的前端与管理框架素材

基于此，当前整体项目更合理的定义是：

> 构建一套面向云边端协同场景的智能算力与物联网管理平台，既要支持算力资源的接入、商品化、交易和运行时访问，也要支持设备、产品、规则、告警、运维和权限等平台化管理能力。

其中：

- `Pivot` 对应平台的“算力交易与运行时控制域”
- `Haihe` 对应平台的“管理运营与业务展示域”

## 3. 总体项目定位

### 3.1 项目性质

该项目可以被理解为一个“平台型系统”，而不是单一业务页面或单一后端服务。其目标不是只完成设备管理，也不是只完成算力交易，而是要形成一个能够支撑资源供给、资源消费、业务运营和系统治理的综合平台。

### 3.2 项目核心特征

项目同时具备以下特征：

- 具备“资源供给侧”能力：算力提供者接入、验收、商品化
- 具备“资源消费侧”能力：算力购买、授权、运行时访问
- 具备“平台管理侧”能力：设备、产品、规则、告警、运维、权限管理
- 具备“控制与安全侧”能力：运行时编排、自研组网或可替换安全接入能力、会话生命周期管理

### 3.3 项目最终形态推断

结合两部分目录的功能边界，整体项目最终更像一个“云边端协同智能平台”，其上层提供平台化管理界面，其下层提供算力资源交易与运行时控制能力。

## 4. 两部分目录的职责划分

### 4.1 `Pivot_Network_Backend-main` 的职责

该目录的核心目标是打通算力交易主链，包括：

- 卖家节点接入
- 节点验收与能力评估
- `Offer` 商品化
- `Order` 下单
- `AccessGrant` 授权
- `RuntimeSession` 创建、恢复与生命周期管理
- 平台控制的安全接入能力
- shell / workspace / task 使用能力

其本质是算力交易与运行时控制的 MVP 原型系统。

### 4.2 `haihe_-main` 的职责

该目录更适合被理解为平台运营侧和业务展示侧的前端与模板集合，包括：

- `ithings-new-main/`：物联网业务前端主体
- `system-template/`：后台权限与管理模板
- `ymir/`：独立参考项目，不宜纳入当前主线

从当前页面和 API 范围看，它主要承担：

- 首页/大屏展示
- 产品管理
- 设备管理
- 分组管理
- 规则引擎/场景联动
- 告警管理
- 运维监控
- 用户/角色/菜单/接口/日志等系统管理

### 4.3 二者关系判断

当前未发现二者之间直接对接代码，因此不宜表述为“已经整合完成的一体化系统”。

更严谨的判断是：

- 二者处于同一大项目的不同层次或不同子域
- `Pivot` 偏平台底座与资源交易
- `Haihe` 偏平台管理前端与业务操作层
- 后续存在整合为统一平台的合理方向

## 5. 总体业务背景

从两个目录共同反映出的需求看，项目试图解决以下几类问题：

### 5.1 算力资源供给问题

- 卖家有闲置算力资源，但缺乏标准接入、验收和售卖机制
- 平台需要将异构节点抽象为可售资源

### 5.2 算力资源消费问题

- 买家希望按需购买算力，并像使用云主机一样使用运行环境
- 平台需要保障授权、隔离、安全访问和会话生命周期控制
- 平台后续组网方案不宜长期依赖 `WireGuard`，需要向自研组网或可替换连接层演进

### 5.3 设备与业务管理问题

- 设备、产品、分组、项目、区域等对象数量多，需要统一管理
- 平台需要具备规则联动、告警闭环、运维监控等常见运营能力

### 5.4 平台治理问题

- 需要统一的账号、角色、菜单、接口和日志管理
- 需要支持后台资源、权限和操作审计

## 6. 项目总体目标

综合两部分内容，项目总体目标可定义为：

### 6.1 平台能力目标

- 建立统一的资源供给与消费平台
- 打通算力节点接入、资源商品化、交易授权、运行时访问主链
- 建立统一的设备与业务管理后台

### 6.2 业务能力目标

- 支持算力提供者与算力消费者两类核心参与方
- 支持设备、产品、规则、告警、运维等业务域管理
- 支持从资源接入到资源使用的完整业务闭环

### 6.3 管理能力目标

- 支持用户、角色、菜单、接口、日志等治理能力
- 支持资源、设备、规则与会话状态的可观测性和可追踪性

## 7. 总体业务角色

### 7.1 算力提供者

- 注册或登录平台
- 接入本地算力节点
- 完成环境准备、加入、验收与状态上报
- 提供可售算力资源

### 7.2 算力消费者

- 浏览可售资源
- 创建订单并获取授权
- 建立运行时会话
- 通过安全通道使用 shell、工作区和任务执行能力

### 7.3 平台管理员

- 管理用户、角色、菜单、接口、日志
- 查看平台资源状态与运行状态
- 参与节点验收、状态纠偏和异常排查

### 7.4 运维人员

- 查看设备运行状态
- 使用在线调试、日志服务和远程配置
- 排查设备、会话和网络链路问题

### 7.5 业务运营人员

- 管理产品、设备、分组、场景联动和告警配置
- 查看平台统计、大屏和业务运行态势

## 8. 总体业务流程

### 8.1 资源供给流程

1. 算力提供者启动本地接入客户端。
2. 平台创建 onboarding session。
3. 提供者完成环境检查、join、probe、join-complete。
4. 平台基于 manager 真相完成验收。
5. 平台执行 capability assessment。
6. 平台生成或更新可售 `Offer`。

### 8.2 资源交易流程

1. 消费者浏览 `Offer`。
2. 创建 `Order`。
3. 平台签发 `AccessGrant`。
4. 消费者通过本地客户端 redeem grant。
5. 平台创建或恢复 `RuntimeSession`。
6. 平台编排运行时、网关和组网接入资源。
7. 消费者通过安全通道进入 shell 并执行任务。

### 8.3 设备管理流程

1. 平台创建产品。
2. 基于产品创建设备。
3. 对设备进行标签、区域、项目和分组组织。
4. 查看设备状态、属性、位置和日志。

### 8.4 规则与告警流程

1. 配置场景联动规则。
2. 设置触发条件和执行动作。
3. 配置告警规则与级别。
4. 告警触发后生成记录与日志。
5. 运维或运营人员根据记录完成处理与追踪。

### 8.5 平台治理流程

1. 管理员创建用户和角色。
2. 配置菜单和接口权限。
3. 系统依据权限返回资源和界面菜单。
4. 平台记录登录日志、操作日志和异常信息。

## 10. 非功能需求

### 10.1 安全性

- 买家不能绕过平台直接访问卖家宿主机
- 平台需要提供统一鉴权与权限边界控制
- 运行时访问需通过授权与安全隧道机制完成

### 10.2 可靠性

- 支持运行时中断后的恢复
- 支持 same-session 重编排后的显式恢复
- 在网络抖动场景下具备可退化、可恢复能力
- Windows 端接入链路需要具备可验证的稳定性基线

### 10.3 可观测性

- 支持节点、会话、任务、规则、设备和告警状态追踪
- 支持日志和错误摘要读取
- 支持业务与基础设施双层状态可视化

### 10.4 可扩展性

- 支持新增设备类型、资源规格和规则类型
- 支持项目、区域、数据权限等企业级扩展
- 支持后续接入更多资源调度或基础设施形态

### 10.5 可维护性

- 明确区分正式目录、模板目录和参考目录
- 保持前后端模型和接口命名一致
- 降低交接难度，提供可复现文档和基础测试说明

### 10.6 跨平台兼容

- 卖家侧重点支持 Windows 本地环境
- 买家侧支持 Linux 与 Windows
- 管理后台应支持常见桌面浏览器访问

## 11. 当前实现现状

### 11.1 `Pivot_Network_Backend-main` 的现状

当前已具备较完整的算力交易 MVP 主链：

- seller 节点接入
- 节点验收与 assessment
- offer / order / grant / runtime session 业务对象
- buyer/seller 本地客户端
- 现有基于 `WireGuard` 的 runtime session 访问链路
- 节点与会话状态查询

但仍存在过渡语义、环境依赖重、测试复现门槛较高等问题。

其中，现有 `WireGuard` 访问链路更适合作为当前原型阶段的过渡实现，不宜继续作为长期正式架构前提。

### 11.2 `haihe_-main` 的现状

当前已具备较完整的物联网管理前端骨架：

- 路由结构完整
- API 模块覆盖面较广
- 设备、产品、规则、告警、运维、系统管理模块齐全

但仍处于明显的演示/联调过渡状态：

- 登录被弱化或跳过
- 菜单和用户资源存在本地模拟
- 请求层默认使用 mock 数据
- 正式业务后端未在仓库内完整体现

### 11.3 两部分当前并未真正整合

当前未发现以下内容：

- 统一网关
- 统一部署脚本
- 统一前后端接口对接关系
- 统一数据库或配置中心

因此，在现阶段不宜将其描述为“已经完成整合的统一系统”。

## 12. 当前主要问题与风险

### 12.1 项目边界分散

- `Pivot` 与 `Haihe` 分别覆盖不同子域
- 目录内部还混有模板与参考项目
- 主开发入口和正式交付范围不够清晰

### 12.2 技术链路不连续

- `Pivot` 强在算力交易和运行时控制
- `Haihe` 强在管理前端和业务展示
- 二者之间缺少明确的正式整合层

### 12.3 前后端成熟度不一致

- `Pivot` 更接近后端和底座原型
- `Haihe` 更接近展示前端和后台骨架
- 两边在“可上线程度”上并不一致

### 12.4 现有组网方案不宜作为长期正式方案

- 当前 `Pivot` 侧访问链路仍大量依赖 `WireGuard`
- 已知 `WireGuard` 组件在 Windows 端稳定性不足
- 后续需要将组网能力抽象出来，并逐步演进为自研组网或可替换连接控制层

### 12.5 需求收口尚未完成

当前仍需确认：

- 最终交付是“算力平台”还是“物联网管理平台”
- 或者最终交付是两者融合后的“云边端协同平台”
- `Haihe` 是否将直接消费 `Pivot` 的部分能力

## 13. 最终项目目标建议

如果需要将两部分统一成一个整体项目，建议将最终目标定义为：

> 建设一套面向云边端协同场景的智能算力与物联网综合管理平台，平台既支持资源供给侧接入与商品化、资源消费侧交易与运行时访问，也支持设备、规则、告警、运维与权限治理等管理能力。

该定义能够同时容纳：

- `Pivot` 的算力交易与运行时能力
- `Haihe` 的平台运营与设备管理能力

## 14. 下一阶段实施建议

### 14.1 第一阶段：明确统一边界

- 明确整体项目最终名称与正式定位
- 明确 `Pivot` 与 `Haihe` 的主从关系或集成关系
- 明确哪些目录属于正式交付范围

### 14.2 第二阶段：确定统一架构

- 明确统一身份认证机制
- 明确统一 API 网关或聚合层
- 明确管理前端与算力后端的对接方式
- 明确资源域与设备域是否共享用户、权限、项目、区域模型
- 明确自研组网层与现有 `WireGuard` 过渡方案的替换路径
- 明确组网控制面由平台统一管理，避免业务主链直接耦合具体隧道组件

### 14.3 第三阶段：打通最小闭环

建议优先打通以下最小闭环：

1. 统一登录
2. 平台管理前端
3. 资源 offer 展示或资源态势展示
4. runtime/session 状态可视化
5. 设备/规则/告警管理

### 14.4 第四阶段：工程化收敛

- 去除 `mock` 与 `placeholder`
- 清理参考目录和历史目录
- 建立最小测试链与部署说明
- 补齐监控、日志和排障手册

## 15. 总结

当前主目录下的两个核心子项目并非简单重复，也不是已经整合完成的同一系统。更合理的理解是：它们分别对应同一大项目的两个子域，一部分解决“算力资源接入、商品化、交易与运行时访问”，另一部分解决“平台运营管理、物联网设备治理与系统后台管理”。

因此，这个项目最终要做的并不是单点功能开发，而是建设一套具备资源交易能力、运行时控制能力、物联网管理能力和平台治理能力的综合平台。现阶段最关键的任务不是继续无边界扩展功能，而是先明确统一定位、统一边界和统一架构，特别是明确自研组网方向，再推进真实对接与工程化收敛。
