# 下一步工作分析报告

更新时间：`2026-05-11`

## 1. 报告目的

基于当前需求梳理结果与现有代码基础，分析项目下一步最合理的推进路径，明确近期工作重点、阶段目标、风险点和决策依据，便于项目组快速进入执行阶段。

## 2. 当前阶段判断

当前项目已经完成了“需求识别”和“项目定位澄清”的前置工作，主要结论如下：

- 当前主线项目应以 `Pivot_Network_Backend-main` 为核心。
- 需求主体与核心接口输入输出已完成初步整理，可作为后续联调和审计依据。
- 项目核心目标不是继续扩展材料，而是尽快验证最小交易主链是否可稳定运行。
- 现有代码不宜直接推倒重做，但也不适合在未经评估的情况下继续堆功能。

因此，下一阶段的重点应从“整理材料”转入“技术收敛与主链验证”。

## 3. 下一步工作的总体原则

### 3.1 先收敛，再扩展

在没有明确主链稳定性之前，不继续大范围增加新功能，否则会放大历史 bug 和架构耦合问题。

### 3.2 先验证核心闭环，再决定修补还是重构

优先围绕“卖家接入 -> 商品化 -> 买家下单 -> 授权 -> 运行时会话”做端到端验证，再决定哪些模块继续修、哪些模块需要重写。

### 3.3 先抽象组网层，再绑定具体实现

由于 `WireGuard` 后续不建议作为长期方案，应先定义组网控制层接口，而不是继续把业务流程直接耦合到现有组件。

### 3.4 先核对接口规格，再进入联调与修复

既然核心接口需求已经明确到输入输出层，下一步应先核对接口文档与现有代码是否一致，再进入联调、修复和重构判断。

## 4. 建议的推进路径

### 4.1 第一阶段：确认正式范围

本阶段目标是明确“什么是正式项目，什么是参考材料”。

完成以下事项：

- 确认 `Pivot_Network_Backend-main` 为当前正式主线。
- 形成一份正式目录与正式交付边界清单。

本阶段输出物应包括：

- 正式目录范围说明
- 当前主线模块清单
- 暂不纳入范围模块清单

### 4.2 第二阶段：主链可运行性审计

本阶段目标是判断现有代码是否还能作为继续开发基础。

围绕以下模块逐项审计：

- 认证与登录链路
- onboarding session 与卖家接入链路
- capability assessment 与 offer 商品化链路
- order / grant / runtime session 交易链路
- 状态查询与异常摘要链路

需要重点确认的问题：

- 哪些接口可以直接运行
- 哪些接口与当前需求文档中的输入输出定义不一致
- 哪些流程仅停留在原型语义
- 哪些模块存在明显的 placeholder 或双轨实现
- 哪些问题属于逻辑 bug，哪些问题属于基础设施依赖问题

本阶段输出物应包括：

- 主链可运行性检查表
- 核心接口对照表
- 已知 bug 清单
- 可修复模块与高风险模块划分

### 4.3 第三阶段：组网与运行时架构收敛

目标是把现有依赖具体组件的实现，收敛为可替换的组网控制架构。

完成以下事项：

- 定义组网控制层接口
- 抽象连接材料、路由、状态、关闭操作等统一能力
- 将 `WireGuard` 降级为过渡实现
- 明确未来自研组网最少要覆盖的能力边界

本阶段输出物应包括：

- 组网控制层设计说明
- 当前 `WireGuard` 依赖点清单
- 替换路径与迁移顺序说明

### 4.4 第四阶段：确定修补或重构策略

在前两阶段完成后，再对项目做修补或重构决策会更稳妥。

采用如下判断标准：

- 如果主链大部分可跑，优先修补并收敛结构。
- 如果关键链路无法稳定复现，且耦合过深，则考虑局部重构。
- 如果组网层、状态层与业务层完全缠绕，则应优先拆层重构，而不是继续打补丁。

## 5. 优先推进的 5 件事

### 5.1 确认正式主线范围

目的：避免多人在不同目录上分散投入。

### 5.2 冻结主链接口与核心对象

目的：把 `Offer`、`Order`、`AccessGrant`、`RuntimeSession` 的真实流转关系和接口输入输出整理清楚。

### 5.3 做一次端到端最小链路验证

目的：快速判断现有代码是“可修”还是“应重构”。

### 5.4 建立 bug 清单与优先级

目的：把“感觉很多 bug”转化为结构化问题列表。

### 5.5 独立定义组网架构

目的：防止后续继续被 `WireGuard` 绑定，影响长期路线。

## 6. 修补与重构的建议判断

当前更推荐的策略是：

> 先做修补性审计，再做局部重构，而不是立即整体推倒重做。

原因如下：

- 当前业务需求已经较清晰，没必要从需求层重新定义。
- 当前已有核心对象和主链原型，具备一定可复用基础。
- 真正的不确定性主要在工程质量、运行稳定性与基础设施耦合，而不是业务方向本身。

但同时也要保留以下判断边界：

- 如果核心主链无法跑通，应及时停止继续叠加开发。
- 如果组网层与业务层难以拆分，应把重构前置。
- 如果环境复现成本过高，应优先建设最小可验证链路。

## 7. 一周工作安排

### 7.1 第 1 天到第 2 天

- 确认正式主线目录
- 整理核心对象与主链流程
- 核对需求文档中的核心接口与现有 API 路由
- 明确汇报口径与开发口径一致

### 7.2 第 3 天到第 4 天

- 逐项检查卖家接入、商品化、订单、授权、会话链路
- 建立接口文档与实际接口行为的差异清单
- 记录可运行接口、报错接口和依赖环境

### 7.3 第 5 天

- 输出 bug 清单
- 输出修补与重构建议
- 输出组网控制层抽象方案初稿

## 8. 当前最重要的决策点

项目下一步最关键的是以下三个决策：

1. 当前正式主线是否锁定为 `Pivot_Network_Backend-main`
2. 现有代码主链是否具备继续修补的价值
3. 组网能力是否从现在开始按可替换控制层来设计

这三个问题如果明确，后续开发方向会明显稳定很多。

## 9. 总结

从当前状态看，项目已经具备进入执行阶段的条件。最合理的推进方式不是继续扩展需求材料，而是围绕主链做一次系统化技术审计，并尽快形成“范围清晰、问题清晰、架构清晰、优先级清晰”的开发起点。

换句话说，下一步最应该做的是：

- 先定边界
- 再验主链
- 再抽组网
- 最后决定修补还是重构
