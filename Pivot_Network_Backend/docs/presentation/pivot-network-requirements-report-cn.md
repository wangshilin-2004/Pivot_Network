# Pivot Network 前后端需求分析与框架可修复性评估

更新时间：`2026-05-10`

## 1. 文档目的

本文用于从需求角度系统描述 `Pivot_Network_Backend-main` 项目应建设的前后端能力，并在文档最后单独给出基于当前已有代码框架的主要问题与可修复性评估。

本文分为两部分：

- 第一部分：仅从产品和业务需求角度定义系统应做什么
- 第二部分：基于当前已有代码框架，对主要问题、风险和修复可行性进行评估

## 2. 项目定义

`Pivot Network` 是一个面向个人算力交易场景的前后端系统，目标是让算力提供者将本地算力节点接入平台并形成可售资源，让算力消费者像使用云主机一样购买、授权并进入运行时环境完成工作。

从用户体验上看，该系统表现为“浏览资源、下单购买、进入远程运行环境并执行任务”；
从系统语义上看，买家实际使用的是平台编排出的 `RuntimeSession`，而不是卖家宿主机本体。

## 3. 建设背景

项目需要解决的核心问题包括：

- 算力提供者拥有闲置算力，但缺少标准化接入、验收和售卖机制
- 算力消费者希望按需购买算力，并获得接近云主机的使用体验
- 平台需要在供给、交易、授权、运行时访问和安全隔离之间建立完整链路
- 平台还需要具备基本的账户、权限、状态追踪和运行管理能力
- 平台后续组网能力不宜长期依赖 `WireGuard`，需要预留自研组网和可替换连接控制能力

因此，该项目不是单一的 Web 页面或单一的后端接口集合，而是一套围绕算力资源供给与消费闭环构建的前后端系统。

## 4. 建设目标

## 4.1 总体目标

建设一套支持算力资源接入、资源商品化、交易授权、运行时访问和状态管理的前后端系统，完成供给侧、消费侧和平台侧的最小业务闭环。

## 4.2 业务目标

- 建立卖家算力节点接入流程
- 建立节点验收和能力评估机制
- 建立可售资源展示与下单流程
- 建立会话授权和运行时环境创建流程
- 建立买家运行时访问、工作区同步和任务执行流程
- 建立自研组网或可替换组网层支持的安全接入能力
- 建立平台账户、权限、状态读取和基础观测能力

## 4.3 交付目标

项目最终应至少实现：

- 一个面向卖家的接入端
- 一个面向买家的消费端
- 一个负责业务真相和控制编排的后端
- 一套可支撑最小端到端演示的资源交易主链

## 5. 业务角色

## 5.1 卖家 Seller

卖家是算力资源提供方，主要职责为：

- 注册或登录平台
- 发起节点接入流程
- 在本地完成环境准备、加入和状态上报
- 提供待验收的算力节点

## 5.2 买家 Buyer

买家是算力资源消费方，主要职责为：

- 注册或登录平台
- 浏览可售资源
- 创建订单并获取访问授权
- 建立运行时会话并进入使用环境

## 5.3 平台管理员 Platform Admin

平台管理员负责：

- 账户与权限管理
- 状态查看与异常排查
- 节点验收辅助与流程纠偏
- 平台运行管理

## 5.4 平台后端

平台后端负责：

- 维护账户、节点、资源、订单、授权和运行时会话的业务真相
- 承担供给侧和消费侧的核心业务编排
- 对接底层运行时和组网控制能力

## 5.5 基础设施控制面

基础设施控制面负责：

- 节点检查
- 运行时创建与销毁
- 组网和安全接入资源编排
- 会话相关运行状态读取

## 6. 总体业务流程

## 6.1 卖家接入流程

1. 卖家登录本地客户端
2. 创建节点接入会话
3. 获取接入材料与指导信息
4. 完成本地环境检查、节点加入与状态上报
5. 平台进行节点验收
6. 节点通过验收后进入可售资源评估流程

## 6.2 资源商品化流程

1. 平台基于通过验收的节点进行能力评估
2. 平台确定可售规格、价格快照和资源描述
3. 平台生成或更新可售 `Offer`
4. 资源进入可被买家浏览和购买的状态

## 6.3 买家交易流程

1. 买家浏览资源列表
2. 买家查看资源详情
3. 买家创建订单
4. 平台激活订单并生成访问授权
5. 买家绑定授权并准备建立运行时会话

## 6.4 运行时使用流程

1. 买家提交会话建立请求
2. 平台创建或恢复 `RuntimeSession`
3. 平台分配连接材料与访问元数据
4. 买家建立平台控制的安全接入通道
5. 买家进入 shell
6. 买家同步工作区
7. 买家执行任务并回读日志

## 6.5 平台观测与管理流程

1. 平台读取节点状态
2. 平台读取资源状态
3. 平台读取订单与授权状态
4. 平台读取运行时会话状态
5. 平台在异常情况下支持排查与纠偏

## 7. 前端需求

## 7.1 卖家前端需求

卖家前端应提供以下能力：

- 登录与注册入口
- 创建 onboarding session
- 展示接入步骤、环境状态和接入结果
- 支持提交接入过程中的探测和完成状态
- 支持自然语言或工具驱动的接入流程
- 展示接入是否通过平台验收

## 7.2 买家前端需求

买家前端应提供以下能力：

- 登录与当前会话状态展示
- 资源浏览与订单关联信息展示
- active grant 拉取与 grant code 导入
- `RuntimeSession` 创建与刷新
- 本地安全接入拉起
- shell 入口展示
- 工作区路径配置与同步入口
- 任务提交与日志回读能力

## 7.3 平台管理前端需求

平台管理端应至少提供：

- 用户状态查看
- 节点状态查看
- 资源状态查看
- 订单和授权状态查看
- 运行时会话状态查看
- 关键异常和最近错误摘要查看

## 8. 后端需求

## 8.1 账户与认证需求

后端应支持：

- 用户注册
- 用户登录
- 用户登出
- 当前用户信息读取
- 会话有效期控制
- 基于角色的访问控制

## 8.2 卖家接入需求

后端应支持：

- onboarding session 创建与读取
- join material 下发
- 多类环境与节点探测数据提交
- join-complete 提交
- correction / override / re-verify 流程
- session heartbeat
- session close

## 8.3 节点验收需求

后端应支持：

- 以平台侧和管理侧状态作为验收依据
- 记录节点识别信息和接入过程数据
- 记录关键网络和节点状态
- 形成验收结果和验收历史

## 8.4 能力评估与商品化需求

后端应支持：

- 基于节点能力生成评估结果
- 确定资源售卖规格
- 自动创建或更新可售 `Offer`
- 在资源不可售时进行下架处理

## 8.5 交易需求

后端应支持：

- 资源列表与详情查询
- 订单创建
- 订单激活
- 访问授权生成
- active grants 查询
- grant id 与 grant code 两种授权绑定方式

## 8.6 运行时会话需求

后端应支持：

- `RuntimeSession` 创建
- `RuntimeSession` 读取
- `RuntimeSession` 刷新
- `RuntimeSession` heartbeat
- `RuntimeSession` stop
- `RuntimeSession` close

## 8.7 组网与安全接入需求

后端应支持：

- 自研组网方案或可替换的连接控制层
- 为运行时会话分配连接材料与接入元数据
- 建立买家访问运行时环境的安全通道
- 支持连接建立、状态读取、异常恢复和主动关闭
- 避免将 `WireGuard` 固化为唯一长期组网方案
- 将组网能力设计为独立抽象层，便于替换底层实现

## 8.8 组网架构约束

结合当前已知限制，项目在组网方向上应明确以下原则：

- 长期正式方案应以自研组网能力为目标
- 当前如继续复用 `WireGuard`，仅可作为过渡实现或参考实现
- Windows 端稳定性应被视为组网方案选型的重要约束
- 业务层、会话层与组网实现层应解耦，避免后续替换时牵动交易主链
- 连接建立、密钥材料、会话路由和状态同步应纳入平台统一控制

## 8.9 文件与下载需求

后端应支持：

- 平台生成文件列表读取
- grant 等文件下载

## 8.10 状态查询需求

后端应支持：

- 节点列表与节点详情查询
- Swarm 总览查询
- 节点过滤检索
- 会话状态与错误摘要读取

## 9. 核心业务对象需求

项目应至少包含以下核心业务对象：

- User
- AuthSession
- SellerOnboardingSession
- CapabilityAssessment
- Offer
- Order
- AccessGrant
- RuntimeSession

上述对象之间应形成清晰关系：

- 卖家通过 onboarding session 接入节点
- 节点通过 assessment 生成 offer
- 买家购买 offer 形成 order
- order 激活后生成 access grant
- access grant 被兑换后生成 runtime session

## 10. 非功能需求

## 10.1 安全性

- 买家不能直接绕过平台访问卖家宿主机
- 访问必须通过授权和平台控制的安全接入链路完成
- 卖家、买家与管理员权限边界必须清晰
- 平台应避免使用单纯本地自报状态作为最终真相

## 10.2 可靠性

- 会话中断后应具备恢复能力
- 网络抖动场景下应尽量保持有限可用
- 运行时状态变化应可追踪、可刷新、可关闭
- Windows 端接入链路应具备可验证的稳定性基线

## 10.3 可观测性

- 平台应能够查看节点状态、资源状态、订单状态和会话状态
- 平台应保留探测、验收、会话与异常相关历史
- 前端应能读到必要的错误摘要与状态信息

## 10.4 可维护性

- 文档应能支撑新成员快速理解主链
- 目录结构应清晰区分正式实现与历史材料
- 核心接口和对象命名应保持统一

## 10.5 跨平台兼容性

- 卖家端重点支持 Windows 本地环境
- 买家端支持 Linux 与 Windows 本地环境
- 平台前端应支持常见桌面浏览器访问

## 11. MVP 交付范围建议

如果当前以最小可交付版本为目标，建议优先保证以下能力完整闭环：

### 11.1 卖家侧 MVP

- 卖家登录
- onboarding session 创建
- 接入材料下发
- probe / join-complete 提交
- 验收状态返回

### 11.2 资源侧 MVP

- capability assessment
- offer 商品化
- offer 列表展示

### 11.3 买家侧 MVP

- 买家登录
- 订单创建
- 授权生成
- runtime session 创建
- 安全接入通道建立
- shell 进入
- workspace sync
- task 执行

### 11.4 平台侧 MVP

- 节点概览
- 会话概览
- 基本状态排查能力

## 12. 基于现有代码框架的主要问题与可修复性评估

本节不再讨论“系统应该做什么”，而是单独评估当前已有代码框架是否适合作为继续开发基础。

## 12.1 评估结论

当前代码框架不属于“完全不可用”的状态，更接近“主链原型已存在，但工程质量不稳定、部分语义处于过渡期”的状态。

因此：

- 不建议直接推倒重做
- 也不建议在未经评估的情况下直接继续堆功能
- 更合理的路径是先做框架级修复，再决定哪些模块保留、哪些模块重写

## 12.2 当前框架的可保留部分

从已有代码结构看，以下部分具有保留价值：

- 业务对象定义基本完整
- 卖家、买家、平台后端三段式结构清晰
- 交易主链已经形成明确建模
- 本地客户端与后端职责边界基本存在
- 会话、授权、资源、验收等概念已经明确

这些内容说明当前项目至少已经完成了业务骨架和主要对象建模，不属于需要从零开始重新定义业务的状态。

## 12.3 当前框架的主要问题

### 12.3.1 业务语义存在过渡实现

当前框架中仍保留明显的过渡语义，例如：

- 部分授权逻辑仍带有 `placeholder` 表述
- 某些运行时材料仍表现为“后续替换”状态

这类问题说明系统业务主链虽然存在，但并未完全从原型态收口到稳定态。

### 12.3.2 存在双轨实现痕迹

当前后端仍保留：

- 内存态存储思路
- 数据库存储思路

这会带来以下问题：

- 理解成本高
- 分支逻辑多
- 状态一致性风险高
- 后续维护时容易引入隐性 bug

### 12.3.3 基础设施依赖较重

当前系统完整运行依赖：

- Docker Swarm
- 现有 `WireGuard` 组网组件
- 运行时控制面
- Windows 卖家环境
- Linux/Windows 买家环境

这意味着项目 bug 很可能不只存在于应用层，也可能存在于环境、网络、编排和状态同步层。

### 12.3.4 当前组网方案与后续需求存在偏差

根据新增约束，后续组网方案不建议继续以 `WireGuard` 作为长期正式方案，主要原因包括：

- `WireGuard` 组件在 Windows 端稳定性不足
- 现有方案受现成组件边界限制较多
- 后续更适合建设可控的自研组网能力

这意味着当前框架中凡是把 `WireGuard` 作为核心固定前提的实现，后续都需要被抽象或替换。

### 12.3.5 历史材料和现行材料混杂

当前仓库中存在：

- 正式目录
- 归档目录
- 备份目录
- 历史说明
- 阶段性脚本

这会显著提高交接和定位成本，也会影响后续判断“当前哪一部分才是真正应维护的实现”。

### 12.3.6 测试和复现门槛仍偏高

虽然仓库中已有测试文件和较完整文档，但当前框架仍存在以下问题：

- 环境搭建成本高
- 本地复现门槛高
- 自动化验证链条不够轻量

这意味着代码可维护性不只取决于逻辑本身，也取决于是否能快速验证修复结果。

## 12.4 可修复性判断

综合判断，当前框架属于“可修复，但需要先做框架级收敛”的类型。

### 适合继续在原框架上推进的前提

- 项目认可当前业务模型基本正确
- 最小交易主链仍可被验证
- 团队希望缩短从需求到演示的时间
- 可接受先做结构收敛再做功能扩展
- 可接受将现有 `WireGuard` 实现降级为过渡方案，并逐步替换为自研组网层

### 更接近需要重构或局部重写的情况

- 如果核心交易对象定义已经不满足目标需求
- 如果主链关键路径无法稳定复现
- 如果基础设施控制层与业务层耦合过深，难以隔离问题
- 如果后续目标已明显超出当前 MVP 设计边界

## 12.5 建议的技术路线

如果决定在现有框架基础上继续推进，建议按以下顺序开展：

1. 先确认正式保留目录和正式主链
2. 收敛 `placeholder` 与双轨实现
3. 将现有 `WireGuard` 依赖抽象为可替换组网层
4. 明确最小端到端验收路径
5. 建立最小可复现测试链
6. 再进入功能扩展和产品化阶段

## 13. 文档结论

从需求角度看，`Pivot Network` 应建设的是一套完整的算力资源交易与运行时访问前后端系统，其核心目标是打通卖家接入、资源商品化、买家购买、访问授权和运行时使用的业务闭环，并逐步形成可控的自研组网与安全接入能力。

从现有代码框架角度看，项目并非空白状态，也并非成熟可直接上线状态，而是具备可继续演进的原型基础。当前更合适的策略不是立即推倒重做，而是先完成框架级问题收敛，尤其需要把 `WireGuard` 从长期架构前提调整为可替换的过渡实现，再依据最小主链的真实可运行性决定保留、重构或局部重写的范围。
