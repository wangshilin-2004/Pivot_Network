# 后端当前状态与接入说明

更新时间：`2026-04-11`

## 1. 当前结论

当前 `Plantform_Backend` 已经和最新 `Seller_Client` 对上了卖家接入主线，不再是“未来要新增 JoinSession”的阶段。

当前阶段定位应理解为：

- `phase3` 末期
  - 核心是验证 seller 节点商品化与平台上架逻辑
- 后续进入 `phase4`
  - buyer `grant-first + RuntimeSession` 设计与实现
- 再进入 `phase5`
  - seller、platform、buyer 闭环联调

现在真实存在并被客户端使用的卖家接入链路是：

1. 卖家在本地 `Seller_Client` 登录或注册 backend 账号
2. 创建 fresh onboarding session
3. backend 向 adapter 请求 `join-material`
4. 卖家本地执行 WireGuard 准备、join、manager task 验证
5. 卖家客户端按 Phase 1 扁平 contract 回写 probes / `join-complete`
6. backend 维护 `manager_acceptance`、`effective_target`、`truth_authority`

## 3. 当前卖家客户端如何使用后端

`Seller_Client/seller_client_app/backend.py` 当前已经固定了后端 API 边界。

客户端会直接依赖这些事实：

- backend 基础地址：`https://pivotcompute.store`
- API 前缀：`/api/v1`
- onboarding response 里必须返回：
  - `swarm_join_material`
  - `required_labels`
  - `expected_wireguard_ip`
  - `manager_acceptance`
  - `effective_target_addr`
  - `effective_target_source`
  - `truth_authority`
  - `minimum_tcp_validation`

此外，客户端当前只接受 Phase 1 的扁平写入 shape：

- `join-complete` 只能写平铺字段
- runtime-local 的 `local_execution / backend_locator / node_probe_summary` 只能放在 `raw_payload`
- backend 不接受把 runtime draft 直接原样嵌套提交

## 4. 当前 join material 与地址真相

当前后端和 adapter 已采用这组权威语义：

- manager 公网地址：`81.70.52.75`
- manager WireGuard / control-plane 地址：`10.66.66.1`
- 当前 seller join target：`10.66.66.1:2377`

也就是说：

- `swarm_join_material.manager_addr` 应返回 `10.66.66.1`
- `swarm_join_command` 应 join 到 `10.66.66.1:2377`
- `81.70.52.75` 只保留公网访问/SSH/外网入口语义

## 5. 当前后端真相链

后端现在维护三组和卖家接入直接相关的状态：

### 5.1 `manager_acceptance`

由 backend 通过 adapter 的 `inspect / claim / inspect` 结果生成。

常见状态：

- `pending`
- `matched`
- `mismatch`
- `node_not_found`
- `inspect_failed`
- `claim_failed`

### 5.2 `effective_target`

当前后端会把会话上的可用目标地址暴露为：

- `effective_target_addr`
- `effective_target_source`
- `truth_authority`

当前优先级是：

1. `manager_matched`
2. `backend_correction`
3. `operator_override`

### 5.3 `minimum_tcp_validation`

当前仍可记录在 session 上，作为服务端附加证据，尤其适用于 correction lane 后的 target 收口。

但它不再取代 seller join 的完成标准。

当前 seller join 的完成标准已经是：

- manager 侧 worker `Ready`
- manager 侧 task 可运行或已运行

## 6. 当前商品化与 buyer 关系

当前 backend 已经能证明两件事：

- 某台 seller 节点接入并被验收通过
- 该节点在 assessment 可售时，已经自动变成 public offer

当前 backend 仍然还没完成的是：

- buyer runtime session 闭环

也就是说，当前状态是：

- seller 节点到商品链已接通
- 商品到 buyer 会话链还没接通

当前 buyer 相关接口仍然停留在交易骨架：

- `Order`
- `AccessGrant`

但 backend 已经能把 onboarding 真相输入到这些 buyer-facing 占位对象里：

- raw manager 已 matched 时，grant 可携带 `manager_matched` target
- backend 已写 `authoritative-effective-target` 时，grant 可携带 `backend_correction` target
- 会话存在 `minimum_tcp_validation` 时，grant 会附带该快照摘要

## 7. 当前未完成的部分

- seller onboarding、auth、trade 都已经迁到数据库
- 当前 onboarding 已拆成规范化子表，并提供 Alembic migration；正式部署应先执行 `alembic upgrade head`
- `SellerNode`、`NodeHeartbeat`、`OfferProfile` 等正式持久化模型还没落库
- runtime bundle 创建/检查/回收还没接进 `orders -> access_grants -> runtime session` 正式链
- manager task execution 的服务端统一记录还没有单独建模在 backend 内
- buyer `grant-first` 与 `RuntimeSession` 还没实现
- `TaskExecution` 还没有正式后端读模型
- 完整 seller -> buyer 闭环还没有跑通

对应的下一步正式设计已经固定在：

- [`Plantform_Backend/docs/卖家节点商品化与能力测算设计.md`](/root/Pivot_network/Plantform_Backend/docs/卖家节点商品化与能力测算设计.md)
