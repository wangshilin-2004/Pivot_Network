# 轻量交易框架借鉴方案

更新时间：`2026-04-07`

## 1. 目标

当前阶段先不实现完整算力交易细节，而是把算力看作一种特殊商品：

- `算力商品 = Offer`
- `下单 = Order`
- `卖家估价 = OfferProfile / Pricing Placeholder`
- `买家凭证 = AccessGrant`

因此，后端完全可以先借鉴轻量 FastAPI 商品/下单框架的模块划分和字段形状，但不直接引入重型商城系统。

本次目标不是“整仓替换成开源商城”，而是：

1. 找一个足够轻量的 `auth + product + order/cart` 借鉴来源
2. 借它的模块边界和字段结构
3. 明确哪些地方必须保留我们自己的平台定制

适合借鉴的地方：

- GitHub 元数据显示 `license` 为空，不适合直接复制代码到主项目里

结论：

- 适合借鉴字段、资源边界、接口命名
- 不适合直接复制代码

## 3. 推荐策略

### 3.1 推荐结论

当前最稳妥的策略不是“整仓拉一个商城项目进来”，而是：

- `认证与用户` 借鉴 `seapagan/fastapi-template`
- `商品 / 购物车 / 商品列表接口形状` 借鉴 `FastAPI-Ecommerce-API`
- `订单 / 卖家估价 / 买家凭证` 保持我们自己定义

### 3.2 为什么这样最合适

因为你要的是：

- 轻量
- 可借鉴
- 不太复杂
- 不需要过多补强和异常处理

而不是：

- 一个现成的重商城
- 一套包含支付、库存、优惠券、物流、后台 CMS 的庞大框架

## 4. 映射到我们平台的方式

### 4.1 可直接套用的电商概念

| 电商概念 | 我们的平台概念 |
| --- | --- |
| `User` | `User` |
| `Product` | `Offer` |
| `Product Category` | `Offer Tier / Runtime Template Category` |
| `Cart` | 可选，占位即可 |
| `Order` | `Order` |
| `Account` | `Seller / Buyer Profile` |

### 4.2 必须保留我们自己定义的概念

| 我们的概念 | 为什么不能直接套商城 |
| --- | --- |
| `JoinSession` | 这是卖家接入流程，不是商城商品流程 |
| `SellerNode` | 节点归属和验收是基础设施问题 |
| `NodeProbe` | 本地环境探测不是电商通用概念 |
| `OfferProfile` | 要承载算力档位、镜像模板、估价输入 |
| `AccessGrant` | 买家访问凭证是我们平台的核心定制 |

### 4.3 需要占位但暂不做实的模块

- `Seller Quote Placeholder`
  - 后续承接卖家估价模块
- `AccessGrant Placeholder`
  - 后续承接买家凭证签发
- `Inventory Placeholder`
  - 后续承接节点是否可售、是否被占用

## 5. 推荐借鉴的最小字段

### 5.1 用户与认证

推荐保留最小字段：

- `User.id`
- `User.email`
- `User.display_name`
- `User.password_hash`
- `User.role`
- `User.status`

### 5.2 商品占位

把算力当商品时，推荐最小字段：

- `Offer.id`
- `Offer.title`
- `Offer.status`
- `Offer.seller_user_id`
- `Offer.seller_node_id`
- `Offer.offer_profile_id`
- `Offer.runtime_image_ref`
- `Offer.price_snapshot`

### 5.3 下单占位

推荐最小字段：

- `Order.id`
- `Order.buyer_user_id`
- `Order.offer_id`
- `Order.status`
- `Order.requested_duration_minutes`
- `Order.price_snapshot`

### 5.4 买家凭证占位

推荐最小字段：

- `AccessGrant.id`
- `AccessGrant.order_id`
- `AccessGrant.buyer_user_id`
- `AccessGrant.status`
- `AccessGrant.grant_type`
- `AccessGrant.connect_material_payload`
- `AccessGrant.expires_at`

### 5.5 卖家估价占位

推荐最小字段：

- `OfferProfile.id`
- `OfferProfile.seller_node_id`
- `OfferProfile.tier`
- `OfferProfile.pricing_inputs`
- `OfferProfile.publishable`

## 6. 当前阶段建议

当前阶段建议直接这样做：

1. `Plantform_Backend` 保持我们自己的项目结构，不整仓引入外部商城。
2. 认证与账户接口风格参考 `seapagan/fastapi-template`。
3. 商品列表、详情、搜索、轻量下单资源风格参考 `FastAPI-Ecommerce-API`。
4. `Order` 先做最小占位，不做复杂支付。
5. `OfferProfile.pricing_inputs` 和 `AccessGrant.connect_material_payload` 先预留好，后续再接卖家估价和买家凭证逻辑。

## 7. 最终建议

如果只选一个“最像我们现在需要借鉴的轻量商品框架”，选：

- `FastAPI-Ecommerce-API`

如果只选一个“最像我们后端长期应该借鉴的认证骨架”，选：

- `seapagan/fastapi-template`

如果问“要不要直接整套搬进来”，当前建议是：

- 不要

原因：

- 直接搬会比自己做一个轻量骨架更乱
- 关键定制点仍然是 `SellerNode / OfferProfile / AccessGrant`
- 这些恰好不是通用电商仓库能替你做好的一层
