# 卖家节点商品化与能力测算设计

更新时间：`2026-04-10`

## 1. 目标

本设计文档用于把当前已经打通的 seller onboarding 真相链，补成一条正式的商品化主线：

- `verified onboarding session -> capability assessment -> offer upsert/downlist`

本轮目标固定为：

- 当 backend 侧 onboarding session 已进入 `verified`，自动对该节点做一次能力测算
- 当测算结果可售时，按 `compute_node_id` 幂等 upsert 一条平台 `listed` offer
- 当测算结果不可售时，如果该节点已有 `listed` offer，则自动下架并保留记录
- 新增一个独立运行的 seller 能力测算接口，支持 seller/admin 以 dry-run 方式查看推荐商品和测算结论

本轮明确不做：

- 不引入完整 `SellerNode` 域模型
- 不做真实 benchmark 调度和任务编排
- 不接受卖家自定义价格
- 不等待卖家镜像上报链，`runtime_image_ref` 直接走平台托管镜像映射

## 2. 当前已确认的 repo 事实

当前代码库里的真实状态如下：

- `backend_app/services/seller_onboarding_service.py` 已经可以把 session 推进到 `verified`
- `verified` 的形成来自 backend 真相链，不来自卖家本地自报成功
- `backend_app/services/trade_service.py` 当前仍然只从 `offers` 表读取 `listed` offer
- `backend_app/repositories/trade_repository.py` 和 `backend_app/storage/memory_store.py` 仍会默认补种 seed offers
- adapter 已暴露：
  - `POST /swarm/nodes/probe`
  - `POST /swarm/runtime-images/validate`
- backend 当前 `AdapterClient` 还没有对应 `nodes/probe` / `runtime-images/validate` 封装

因此当前 repo 的真实情况仍然是：

- onboarding 真相链已经存在
- `/offers` 商品链仍是占位交易面
- “verified node 自动变成真实商品”这一步还没有落地

## 3. 主流程定义

### 3.1 自动商品化主线

自动商品化流程固定为：

1. seller onboarding session 在 backend 内被写成 `verified`
2. backend 触发内部 capability assessment
3. assessment 解析 target、收集证据、补 adapter probe、做 runtime image validate
4. assessment 按平台规则生成推荐 offer 结果
5. 若 assessment 可售，则按 `compute_node_id` upsert offer 并置为 `listed`
6. 若 assessment 不可售，且该节点已有 `listed` offer，则自动下架

关键约束：

- onboarding 的 `verified` 是独立真相，不会因为 assessment 或 offer upsert 失败而被回滚
- 商品化是 `verified` 之后的后置副作用
- 任何自动商品化失败都只能影响商品链，不能改写 onboarding 真相链

### 3.2 哪些入口触发自动商品化

所有会把 session 最终写成 `verified` 的后端路径，都必须在提交 onboarding 真相后触发内部商品化：

- `submit_join_complete`
- `reverify_manager_acceptance`
- `submit_minimum_tcp_validation`

触发规则固定为：

- 只要本次请求结束时 session 状态为 `verified`，就执行一次 assessment + offer apply
- 不要求“第一次变成 verified”才触发；重复 verified 也要触发，用于刷新 offer

## 4. 服务拆分

### 4.1 `CapabilityAssessmentService`

新增独立服务，职责固定为：

- 解析 assessment target
- 收集 onboarding / adapter / seller supplemental 三类证据
- 生成 `measured_capabilities`
- 生成 `pricing_decision`
- 选择平台托管 `runtime_image_ref`
- 调 adapter 执行 `runtime-images/validate`
- 产出 `recommended_offer`
- 持久化 `seller_capability_assessments`

该服务需要支持两种模式：

- 外部 dry-run：持久化 assessment，但不写 `offers`
- 内部 apply-offer：复用同一套 assessment 逻辑，并把结果交给商品化服务

### 4.2 `OfferCommercializationService`

新增独立服务，职责固定为：

- 接收 assessment 结果
- 按 `compute_node_id` 查找现有 offer
- 在可售时幂等 upsert offer
- 在不可售时自动下架已有 `listed` offer
- 维护 `source_join_session_id`、`source_assessment_id`、`inventory_state`

### 4.3 服务边界

`SellerOnboardingService` 不直接实现测算规则与 offer 写入细节，它只负责：

- 完成 onboarding 真相写入
- 在 session 落为 `verified` 后调用内部商品化入口

`TradeService` 继续负责 buyer-facing `/offers` / `orders` / `access_grants` 读写，但真实商品数据来源将从 seed/demo offer 迁移到 assessment 自动生成 offer。

## 5. 独立能力测算接口

### 5.1 路由与鉴权

新增路由：

- `POST /api/v1/seller/capability-assessments`

允许角色：

- `seller`
- `platform_admin`

### 5.2 请求体

请求固定采用单次测算模型：

```json
{
  "onboarding_session_id": "join_session_xxx",
  "compute_node_id": "compute-seller-1",
  "node_ref": "node-abcd",
  "requested_offer_tier": "medium",
  "requested_accelerator": "gpu",
  "seller_reported_capabilities": {},
  "seller_reported_benchmarks": {}
}
```

约束固定为：

- `onboarding_session_id` 与 `compute_node_id` 二选一，至少提供一个
- `node_ref` 仅作为可选补充定位字段
- `requested_offer_tier`、`requested_accelerator` 可覆盖 assessment 本次推导
- `seller_reported_capabilities` / `seller_reported_benchmarks` 只作为补充证据，不覆盖平台真相

### 5.3 target resolution

assessment target 解析规则固定为：

1. 如果传了 `onboarding_session_id`，直接以该 session 为准
2. 如果只传 `compute_node_id`：
   - `seller` 调用：先取当前 seller 名下该 `compute_node_id` 的最新 `verified` session，没有再退到该 seller 名下最新 session
   - `platform_admin` 调用：先取该 `compute_node_id` 的全局最新 `verified` session，没有再退到全局最新 session
3. 如果 `compute_node_id` 对应多个 seller 且 admin 无法唯一归属，则返回 `409`，要求显式改传 `onboarding_session_id`
4. `node_ref` 只作为 adapter inspect/probe 的补充，不单独决定 ownership

resolved target 输出固定包含：

- `onboarding_session_id`
- `seller_user_id`
- `compute_node_id`
- `node_ref`

### 5.4 响应体

响应固定包含：

- `assessment_id`
- `assessment_status`
- `resolved_target`
- `sources_used`
- `measured_capabilities`
- `pricing_decision`
- `runtime_image_validation`
- `recommended_offer`
- `warnings`

`assessment_status` 固定使用以下枚举：

- `sellable`
- `node_not_found`
- `probe_failed`
- `runtime_image_invalid`
- `insufficient_evidence`

其中只有 `sellable` 代表可售；其余状态都视为不可售。

### 5.5 外部接口行为

外部 `POST /api/v1/seller/capability-assessments` 固定行为为：

- 持久化一条 assessment 记录
- 返回 assessment 结果与推荐 offer
- 不直接改写 `offers`

## 6. 证据来源与真相优先级

### 6.1 source-of-truth 规则

assessment 的判断规则固定为：

- 节点是否存在，以 adapter 当前查到的节点为准
- 节点当前能力，以 backend 已存 probe 为基础，以 adapter 当前 probe 做二次校正
- runtime image 是否可运行，以 adapter `runtime-images/validate` 当前结果为准

### 6.2 证据优先级

能力字段填充顺序固定为：

1. backend 已存 onboarding probe
   - `probe_summary.resource_summary`
   - `linux_substrate_probe`
2. adapter `POST /swarm/nodes/probe`
3. seller supplemental payload

使用语义固定为：

- onboarding probe 用于补全 seller 本地采集事实
- adapter probe 用于确认当前 Swarm 节点容量与实时可见性
- seller supplemental 只进 `sources_used`、assessment request snapshot、warnings
- seller supplemental 不能直接覆盖平台真相

### 6.3 不可售判定

以下任一条件命中时，assessment 一律不可售：

- adapter 节点不存在
- adapter `probe_status != probed`
- runtime image validate 返回 `validation_status != validated`
- 无法解析出最低可售能力信息，例如 CPU / 内存都缺失

## 7. 测算规则、档位与定价

### 7.1 v1 lane 规则

v1 先固定两条 lane：

| accelerator | default tier | offer_profile_id | runtime_image_ref | price_snapshot |
| --- | --- | --- | --- | --- |
| `gpu` | `medium` | `profile-medium-gpu` | `registry.example.com/pivot/runtime:python-gpu-v1` | `{"currency":"CNY","hourly_price":12.5}` |
| `cpu` | `small` | `profile-small-cpu` | `registry.example.com/pivot/runtime:python-cpu-v1` | `{"currency":"CNY","hourly_price":4.0}` |

### 7.2 accelerator 解析

accelerator 解析优先级固定为：

1. request 的 `requested_accelerator`
2. session 的 `requested_accelerator`
3. onboarding probe 的 `gpu_available`
4. adapter probe 的 `accelerator_label`
5. 默认回退为 `cpu`

### 7.3 tier 解析

tier 解析固定为：

1. request 的 `requested_offer_tier`
2. session 的 `requested_offer_tier`
3. lane 默认 tier

但 v1 只支持当前规则表里的 profile：

- `gpu -> medium`
- `cpu -> small`

如果请求了当前不支持的 tier：

- 不报错
- 回退到该 lane 默认 tier
- 在 `warnings` 里加入 `requested_offer_tier_unsupported_fell_back_to_default_lane`

### 7.4 `measured_capabilities`

assessment 的标准化输出至少包含：

- `cpu`
- `memory`
- `gpu`
- `accelerator`
- `generic_resources`
- `warnings`

建议 shape：

```json
{
  "cpu": {
    "logical": 8,
    "source": "adapter_probe"
  },
  "memory": {
    "total_mb": 32768,
    "source": "adapter_probe"
  },
  "gpu": {
    "available": true,
    "source": "onboarding_probe"
  },
  "accelerator": "gpu",
  "generic_resources": [],
  "warnings": []
}
```

### 7.5 `capability_summary`

写入 offer 的 `capability_summary` 固定是 `measured_capabilities` 的 buyer-facing 摘要，不直接存整份原始 probe：

- `cpu_limit`
- `memory_limit_gb`
- `gpu_mode`
- `accelerator`
- `generic_resources`
- `warnings`

### 7.6 `pricing_decision`

`pricing_decision` 固定包含：

- `rule_version`
- `resolved_accelerator`
- `resolved_offer_tier`
- `offer_profile_id`
- `price_snapshot`

卖家不能手填或覆盖 `price_snapshot`。

## 8. runtime image validate

v1 `runtime_image_ref` 不等待卖家镜像链，固定走平台托管镜像映射。

validation 规则固定为：

1. 先按 lane 选出 `runtime_image_ref`
2. 调 adapter `POST /swarm/runtime-images/validate`
3. 只有 `validation_status == validated` 才允许进入 `sellable`

`runtime_image_validation` 输出至少包含：

- `image_ref`
- `validation_status`
- `checks`
- `validation_payload`

如果 validate 失败：

- assessment 返回 `runtime_image_invalid`
- `recommended_offer.publishable = false`
- 若已有 `listed` offer，则自动下架

## 9. Offer upsert / downlist 语义

### 9.1 唯一性

`offers.compute_node_id` 必须建立唯一索引。

含义固定为：

- 一台节点在平台内只保留一条当前 offer 记录
- 不按每次 onboarding 新建重复商品

### 9.2 创建与更新

当 assessment 可售时：

- 按 `compute_node_id` 查找现有 offer
- 不存在则新建
- 存在则更新

更新字段固定为：

- `title`
- `status`
- `seller_user_id`
- `seller_node_id`
- `compute_node_id`
- `offer_profile_id`
- `runtime_image_ref`
- `price_snapshot`
- `capability_summary`
- `inventory_state`
- `source_join_session_id`
- `source_assessment_id`
- `updated_at`

字段规则固定为：

- `status = listed`
- `seller_node_id` v1 直接镜像 `compute_node_id`
- `published_at` 仅在第一次成功上架时写入，后续更新保留原值
- `title` 采用稳定规则：`{resolved tier title} {resolved accelerator} Runtime ({compute_node_id})`

### 9.3 自动下架

当 assessment 不可售且已存在 `listed` offer 时：

- 不删除 offer
- 把 `status` 从 `listed` 移出，建议设为 `unavailable`
- 更新 `inventory_state`
- 更新 `source_assessment_id`
- 更新 `updated_at`

这样 `/offers` 仍只返回当前可售节点，但历史 offer 记录还保留在库里。

### 9.4 `inventory_state`

`inventory_state` 固定至少包含：

- `available`
- `assessment_status`
- `verified_source`
- `assessment_id`
- `reason`

推荐 shape：

```json
{
  "available": true,
  "assessment_status": "sellable",
  "verified_source": "seller_onboarding_verified",
  "assessment_id": "assessment_xxx",
  "reason": null
}
```

## 11. seed/demo offer 策略

当前 seed offers 仍存在于 repo，用于本地开发和测试。

但在新主线里，seed/demo offer 不再作为正式 public listing 依赖：

- 自动生成的真实 offer 才是 `/offers` 主来源
- seed offer 只能通过显式开发/测试入口补种
- 不能继续在正式链路里默认补种成 public `listed` 数据

这意味着后续实现需要同步收口：

- `TradeRepository.ensure_seed_offers()`
- `InMemoryStore.seed_offers()`
- Alembic `0002_auth_trade_core.py` 的 `_seed_offers`

它们要么改成开发态显式开关，要么仅在测试夹具里使用。

## 13. 测试与验收

必须覆盖的测试场景：

- onboarding session `verified` 后，自动 assessment + offer upsert 创建真实 `listed` offer
- 同一 `compute_node_id` 二次 verified / 重测只更新同一条 offer，不生成重复项
- 独立 assessment 接口默认只写 assessment，不直接写 `offers`
- 只有 onboarding probe
- onboarding probe + adapter probe
- onboarding probe + adapter probe + seller supplemental
- adapter 节点不存在时返回不可售
- adapter `probe_failed` 时返回不可售
- runtime image validate 失败时返回不可售并自动下架已有 listing
- `/offers` 返回真实自动生成 offer，且 `compute_node_id`、`capability_summary`、`price_snapshot` 正确
- Alembic migration 可以创建 assessment 表并扩展 `offers`
- 现有 auth / onboarding / trade 回归继续通过

额外回归要求：

- trade 相关测试不再隐式依赖默认 seed offers
- 需要改成测试中显式构造真实 offer 或显式打开 seed fixture

## 14. 默认值与结论

本设计固定采用以下默认值：

- v1 不引入完整 `SellerNode` 域模型
- v1 不做真实 benchmark 调度，只做 probe-based placeholder assessment
- `session.status == verified` 仍是自动商品化入口
- assessment 失败时自动下架已有 listing，而不是继续 public listed
- `compute_node_id` 解析 session 时，优先最新 `verified`，没有再回退最新 session

这份文档的定位是实现规格，不是现状说明。

README 与“后端当前状态”文档只负责说明：

- 当前已经做到什么
- 当前还没做到什么
- 下一步的正式设计已经定在本文件
